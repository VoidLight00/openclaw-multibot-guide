1. 개요

왜 하나의 OpenClaw로 여러 에이전트를 운영해야 할까요? 멀티 에이전트 아키텍처의 장점과 핵심 개념을 알아봅니다.

🤔 왜 멀티 에이전트인가?

"하나의 AI 비서로 모든 걸 하면 되지 않나요?"라고 생각할 수 있습니다. 하지만 실제로 사용해보면 여러 에이전트로 분리하는 것이 훨씬 효율적입니다.

단일 에이전트의 한계

[X] 컨텍스트 오염

트레이딩 분석 중에 "오늘 저녁 뭐 먹지?" 같은 대화가 섞이면 맥락이 깨집니다.

[X] 페르소나 충돌

전문적인 코딩 어시스턴트가 갑자기 친근한 말투로 바뀌면 어색합니다.

[X] 모델 비용

간단한 리마인더에도 비싼 Claude Opus를 쓰는 건 낭비입니다.

멀티 에이전트의 장점

[OK] 역할 분리

각 에이전트가 전문 분야에 집중. 트레이딩봇, 코딩봇, 일상봇 분리.

[OK] 컨텍스트 격리

대화 히스토리가 섞이지 않아 더 정확한 응답.

[OK] 모델 최적화

복잡한 작업엔 Opus, 간단한 대화엔 Flash. 비용 절감.

[OK] 독립적 설정

에이전트마다 다른 도구, 메모리, 성격 설정 가능.

[BUILD] 아키텍처

OpenClaw 멀티봇의 핵심 아키텍처를 이해해봅시다.

핵심 컴포넌트

컴포넌트 역할 설명
Gateway 중앙 라우터 모든 메시지를 수신하고 적절한 에이전트로 전달
Agent AI 비서 독립적인 설정, 메모리, AI 모델을 가진 개별 비서
Channel 메시징 연결 Telegram, Discord 등 외부 메신저와의 연결
Binding 라우팅 규칙 어떤 채널의 메시지를 어떤 에이전트로 보낼지 결정
Workspace 작업 공간 에이전트의 파일, 메모리, 설정이 저장되는 디렉토리

메시지 흐름

메시지 처리 과정 
1. 사용자가 @MyBot_bot 에 "안녕" 전송
                    │
                    ▼
2. Telegram API가 OpenClaw Gateway로 메시지 전달
                    │
                    ▼
3. Gateway가 bindings 확인
   → "telegram + MyBot_bot = Agent A"
                    │
                    ▼
4. Agent A가 메시지 처리
   → workspace 컨텍스트 로드
   → AI 모델 호출 (Claude/Gemini/GPT)
   → 응답 생성
                    │
                    ▼
5. 응답이 Telegram으로 전송
